
# Enterprise

# USAGE GUIDELINES

- Use buttons when explicit user interaction is required.

- Lead with an actionable verb that clearly reflects the next step (e.g. Buy Now, Save Flight, Add to Cart).

- Button labels should be concise, with a maximum of 4 words.

- Use Title Case for button labels.

- Avoid using buttons for decorative or non-interactive elements.

- Use only one Primary button per screen to maintain clear action hierarchy.

- Button width may either fill the page container or adapt to the text length. The choice is defined by the designer based on project requirements.

- Minimum button width on Desktop is 240px** **to ensure tap target and visual consistency.
# CONTENT GUIDELINES

- Use clear, action-oriented verbs (e.g. Save, Pay, Continue).

- Keep labels short to avoid truncation.

- Maintain consistent verb tense across buttons on the same screen.
# BEHAVIOUR (INTERACTIONS)

  

### **Interaction Feedback**
- Provides immediate visual response on user interaction.
- Disabled state blocks all interaction and feedback.

### Validation & Error Handling
- Loading state is triggered when an action initiates an asynchronous process.
- Button remains visible but non-interactive during loading.
- No duplicate submissions until process completes.

### **Swipe Behaviour**
- Requires deliberate, complete swipe
- Partial swipe does not trigger action
